# Plan — Global Summarization Models + Compaction Robustness

Status: **IMPLEMENTED** (2026-06-01). Project is **unreleased** (per CLAUDE.md) — no
backwards-compatibility shims; deprecated config removed rather than retained.

Confirmations resolved: (1) `compress_model` deleted entirely, load-time WARN for
legacy files; (2) `ForceCompress` (413 recovery) bypasses the breaker; (3) failure
counter is in-memory only and resets on restart.

## Goals (decisions locked with user)

1. **Global summarization model list only.** Remove per-agent `compress_model`
   (both `agents.*.compress_model` and `agents.defaults.compress_model`). One
   ordered, optional global list drives summarization for every agent.
2. **Append agent primary as last resort.** Chain = `[global summarization models…, agent primary]`.
3. **3-strike circuit breaker.** After 3 consecutive failed *automatic* compactions
   for a session, suppress auto/cron compaction for a cooldown. A user `/compact`
   always bypasses the breaker.
4. Distinguish the conflated error states ("no client" / "nothing to compress" /
   "all models rejected").
5. Make a validation-rejected summary fall through to the next model in the chain.
6. WebUI exposes the global list (add / remove / reorder), no per-agent field.

Out of scope (explicitly deferred by user): model-specific prompt hardening for
models that can't satisfy the cited-refs requirement (e.g. grok-class).

---

## Root causes being fixed (from log investigation)

- **Mode 1 — nothing to summarize (instant):** retained tail already covers the
  whole conversation → `toSummarize` empty → loop breaks → `llmSucceeded=false` →
  `ErrCompressionFailed`. Surfaced as "Failed to compact history" on an
  already-compact session (dawn 10:51).
- **Mode 2 — every summary rejected (slow):** LLM is called up to 6×
  (3 standard + 3 aggressive), each summary fails `HasMaterial`/`HasEvidence`,
  `llmSucceeded` never flips → same error. Fired hourly via cron for days
  (dawn 05‑27→05‑30; sam/grok now). Wasted ~6 calls per trigger, no backoff.
- **Fallback never engaged on validation failure:** `callLLMChain` returns
  `ok=true` as soon as one client returns *parseable* JSON; the
  `HasMaterial`/`HasEvidence` check runs *outside* the chain, so a rejected summary
  re-tries the **same** model instead of advancing to the fallback.
- **`compress_model.fallbacks` is dead config:** declared in `AgentModelConfig` but
  never read by the compression path.

## llmcontext loop changes (`pkg/llmcontext/compress.go`, `errors.go`, `manager.go`)

### 1. Distinct errors (`errors.go`)

```go
ErrCompressionFailed   = errors.New("llmcontext: compression failed")        // attempts made, all rejected
ErrNothingToCompress   = errors.New("llmcontext: nothing to compress")       // tail already covers everything
// (no client configured stays a silent no-op returning nil, as today)
```

### 2. Move validation into the client chain (`callLLMChain`)

Push the post-parse checks (`StripOutOfRangeSeqRefs`,
`HasMaterial`/`HasEvidence`, max-summary-token truncation + oversize discard)
**inside** the per-client loop in `callLLMChain`. A client that returns a
parseable-but-invalid summary now `continue`s to the **next** client instead of
returning `ok=true`. `callLLMChain` returns `ok=true` only for a fully valid,
fitted summary. This makes the `[global list…, primary]` chain actually exercise
fallback on Mode‑2 failures.

`callLLMChain` will need `summaryTokenLimit`, `contextWindow`, and `sessionKey`
passed in (for the truncation budget and existing WARN logs).

### 3. No-op vs failure (`doCompress` + `handleNormalPostLoop`)

Track two flags in the loop:
- `attemptedLLM` — set true the first time `callLLMChain` is invoked with a
  non-empty `toSummarize`.
- `llmSucceeded` — unchanged.

Post-loop classification (normal path):
- `llmSucceeded` → persist, success (unchanged).
- `!llmSucceeded && !attemptedLLM` → **nothing to compress** → return
  `ErrNothingToCompress`. Does **not** count as a breaker failure.
- `!llmSucceeded && attemptedLLM` → all models rejected → return
  `ErrCompressionFailed`. **Counts** as a breaker failure.

Replace the single misleading WARN with messages keyed to the case.

### 4. Circuit breaker (`manager.go`)

Add to `Manager`:
```go
consecutiveCompactFailures int
breakerTrippedUntilCount   int   // msgCount until which auto-compaction is suppressed
```

- The auto path is `triggerCheck → compress(safetyNet) → doCompress`. The
  *manual* path is `Compact → doCompress`. Thread a `manual bool` (or a small
  internal mode enum) so the breaker only gates the auto path.
- In `compress()` (auto): if breaker tripped (`msgCount < breakerTrippedUntilCount`),
  skip with an INFO log and return nil.
- After `doCompress` on the auto path:
  - `ErrCompressionFailed` → `consecutiveCompactFailures++`; at `>= 3`, set
    `breakerTrippedUntilCount = msgCount + cooldownMessages` (reuse/define a const,
    e.g. 20) and log a WARN naming the session + the models tried.
  - success or `ErrNothingToCompress` → reset `consecutiveCompactFailures = 0`.
- `Compact()` (manual): bypasses the breaker check entirely, and a successful
  manual run resets the counter. Treat `ErrNothingToCompress` as success.
- Decide for the 413-recovery path (`ForceCompress`, loop.go:1736): it is a real
  context-overflow recovery, so it **bypasses** the breaker like manual. Note in
  code comment.
- In-memory is acceptable (resets on restart). Optional: persist
  `consecutiveCompactFailures` in the existing compaction-state blob so an
  hourly-failing cron doesn't reset every restart — include if cheap.
